# Module 8 Journal CS255

The DriverPass project was based on a client who wanted a system designed for a driver training business. The client, DriverPass, wanted to help students prepare for their driving tests by giving them access to online classes, practice tests, and the ability to schedule on-road driving lessons with instructors. The system needed to support different users, including customers/students, instructors, the secretary, the IT officer, and the owner. Each user needed access to different parts of the system depending on their role. Overall, the goal was to design a system that made scheduling, training, package management, and user access easier to manage in one place.

One thing I think I did particularly well was breaking down the client’s needs into specific system requirements. Instead of only looking at the system from a technical point of view, I tried to think about what each user would actually need to do. For example, students needed to be able to create an account, log in, purchase packages, schedule lessons, and take practice tests. Instructors needed to view their schedules and leave notes about student progress. The secretary needed to manage appointments and customer information, while the owner and IT officer needed more administrative access. I think I did a good job connecting those needs back to the diagrams and requirements.

If I could revise one part of my work, I would improve the UML diagrams and make sure the relationships and symbols are more accurate. I understood the overall process, but some of the diagram details could have been clearer, especially the include relationships in the use case diagram and the way alternative paths were shown in the sequence diagram. I would go back and make sure each actor only has access to the correct use cases, and I would clean up the connections so the diagrams better show how the system works. This would make the design easier for both technical and nontechnical people to understand.

I interpreted the user’s needs by reviewing the DriverPass interview and identifying what the client was really asking for. I tried to separate the system into different user roles and think through what each role needed to accomplish. Those needs were then turned into functional requirements, nonfunctional requirements, and UML diagrams. Considering the user’s needs is important because the system is not just being built to exist; it is being built to solve a real problem. If the design does not match what the users need, then the system may be technically complete but still frustrating or ineffective to use.

My approach to designing software is to start with the problem first, then move into the technical design. I like to understand who the users are, what they need, and what the system has to accomplish before thinking about diagrams or implementation details. In the future, I would continue using interviews, requirements documents, use case diagrams, activity diagrams, sequence diagrams, and class diagrams to analyze and design systems. I would also spend more time reviewing the diagrams for accuracy, because small notation issues can change how the design is understood. Overall, this project helped me see how important it is to connect customer requirements to a clear system design.
